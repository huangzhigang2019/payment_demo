## User Scenarios & Testing *(mandatory)*

### User Story 1 - 完成一笔“消费/支付”交易（Priority: P1）

作为测试/演示人员，我希望在设备上输入交易金额并发起交易，系统引导我完成读卡、必要的交互（如密码输入/持卡人确认/重试），并最终展示交易结果（成功/失败/取消/超时），从而可端到端演示“从 0 到 1 的交易流程”。

**Why this priority**: 这是 Demo 应用的核心价值，能最直接验证交易主流程与关键交互闭环。

**Independent Test**: 仅实现此故事，也能通过“输入金额 → 发起交易 → 读卡 → 展示结果”完整演示并验证端到端流程。

**Acceptance Scenarios**:

1. **Given** 应用已启动且设备处于可交易状态，**When** 用户输入合法金额并点击“开始交易”，**Then** 系统进入引导读卡界面并开始等待卡片。
2. **Given** 处于等待读卡中，**When** 成功读取到卡片并完成必要的交易处理，**Then** 系统展示“交易成功”及关键摘要信息（至少包含金额、时间、交易结果码/原因）。
3. **Given** 处于等待读卡或交易处理中，**When** 用户点击“取消”，**Then** 系统中止流程并展示“已取消”，且可返回发起交易页面再次开始。
4. **Given** 处于等待读卡中，**When** 超过设定等待时间仍未读到卡，**Then** 系统提示“超时”并允许重试或返回。

---

### User Story 2 - 支持多种读卡方式与回退提示（Priority: P2）

作为测试/演示人员，我希望系统能识别并支持常见读卡方式（接触式/非接触式/磁条，如设备支持），并在某种方式失败或不支持时给出清晰提示（例如改用另一种方式或重新刷卡），从而覆盖更多现场演示场景。

**Why this priority**: 提升 Demo 覆盖面与可演示性，减少因环境/卡片差异导致的演示失败。

**Independent Test**: 可通过分别尝试不同介质的卡片完成测试，验证提示、分支流程与结果展示。

**Acceptance Scenarios**:

1. **Given** 设备支持非接触式读卡，**When** 用户挥卡，**Then** 系统识别为非接触式并继续处理直到展示结果。
2. **Given** 某读卡方式不支持或读卡失败，**When** 系统判定不可继续，**Then** 系统给出可操作的提示（重试/换方式/取消）并能回到可继续的状态。

---

### User Story 3 - 可查看基础信息与必要配置状态（Priority: P3）

作为开发/测试人员，我希望能在应用内查看与交易相关的基础信息（版本信息、关键配置是否就绪、必要参数是否加载），以便快速定位“为什么交易无法开始/无法完成”的原因。

**Why this priority**: 降低联调与演示成本，提升排障效率。

**Independent Test**: 在不插卡的情况下即可验证信息展示的完整性与可读性。

**Acceptance Scenarios**:

1. **Given** 应用已安装启动，**When** 用户进入“设置/信息”页面，**Then** 系统展示版本信息与关键状态（例如：参数是否已准备、读卡模块是否可用）。

---

### Edge Cases

- 读卡过程中卡片移开/拔出，系统应提示并允许重试或取消。
- 交易处理中发生异常/不可恢复错误，系统应结束流程并展示明确失败原因（面向测试人员可读），且应用不崩溃。
- 交易交互（如确认/输入）长时间无操作，系统应超时并回到可重试状态。
- 金额输入非法（为空、0、超出上限、格式错误），系统应阻止开始交易并提示修正。

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 系统 MUST 提供“发起交易”的入口，并支持输入交易金额（含基本校验与错误提示）。
- **FR-002**: 系统 MUST 在交易开始后引导用户完成读卡，并在读卡成功后进入交易处理流程。
- **FR-003**: 系统 MUST 在交易过程中支持用户取消，并保证取消后能回到可再次发起交易的状态。
- **FR-004**: 系统 MUST 对“超时”进行处理（至少包含：等待读卡超时、关键交互超时），并提供重试/返回的用户路径。
- **FR-005**: 系统 MUST 在交易结束时展示结果页，结果页 MUST 至少包含：交易金额、交易时间、结果状态（成功/失败/取消/超时）以及可定位问题的原因摘要。
- **FR-006**: 系统 MUST 支持在应用内查看基础信息与关键配置状态（用于开发/测试快速判断是否具备交易条件）。
- **FR-007**: 系统 SHOULD 支持多种读卡方式（若设备能力支持），并在不支持/失败时提供可操作提示与回退路径。
- **FR-008**: 系统 MUST 使用项目提供的 UI 资源文件呈现核心页面（发起交易、读卡引导、处理中、结果、设置/信息），确保视觉与交互一致性。

### Key Entities *(include if feature involves data)*

- **Transaction**: 一笔交易的抽象；关键属性包括金额、开始/结束时间、介质类型、结果状态、失败原因摘要、参考号/流水号（如可得）。
- **Card Read Result**: 读卡阶段的结果；关键属性包括介质类型、是否成功、失败原因、必要的卡片标识信息（如可得）。
- **Processing Result**: 交易处理结果；关键属性包括结果状态、结果码/原因、需要展示给用户的摘要信息。
- **Terminal/Config Status**: 终端与配置就绪状态；关键属性包括“是否可交易”“关键参数是否加载”“版本信息”等。

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 在实验/演示环境中，用户从“开始交易”到看到“结果页”的时间在 60 秒内完成（不含用户人为停顿）。
- **SC-002**: 在 20 次连续演示中，应用崩溃次数为 0（Crash-free）。
- **SC-003**: 在提供的标准测试卡与设备环境下，P1 主流程首轮完成率达到 90% 以上（失败需有明确原因展示）。
- **SC-004**: 对于取消与超时场景，用户在 3 步以内能回到“可再次发起交易”的状态。

## Assumptions

- 该 Demo 面向内部开发/测试/演示人员使用，允许展示更偏“可定位问题”的错误摘要信息。
- 交易类型以“消费/支付”为主，其他交易类型（如撤销/退货等）不作为本次最小可交付范围的强制项。
- 设备读卡与交易处理能力由公司内部提供的能力/库提供；本规格不限定具体技术实现，仅要求对用户可见行为与结果满足上述需求。

## Dependencies

- 读卡与基础设备能力依赖公司内部设备能力接口（例如 NeptuneLiteAPI 所代表的能力集合）。
- EMV 卡处理流程依赖项目提供的 EMV 相关库及其推荐流程（可参考 `Reference_Project/JemvDemo2.0` 的业务流）。
- UI 视觉与交互资源依赖 `UI_Resource/` 提供的资源文件。
